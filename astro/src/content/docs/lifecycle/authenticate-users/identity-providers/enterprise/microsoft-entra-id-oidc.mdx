### Register a New Microsoft Entra ID Application

You will first need to log in to the [Microsoft Entra admin center](https://entra.microsoft.com/) (or the Azure portal).

Navigate to <Breadcrumb>Identity -> Applications -> App registrations</Breadcrumb> and click <InlineUIElement>+ New registration</InlineUIElement> to create a new Microsoft Entra ID Application.

![Register a new Microsoft Entra ID Application](/img/docs/lifecycle/authenticate-users/identity-providers/enterprise/azure-openid-connect-register-application.png)

Here we have configured our application `Redirect URI`.  If FusionAuth is running at `https://local.fusionauth.io`, this value should be `https://local.fusionauth.io/oauth2/callback`.

![Microsoft Entra ID Client ID and Tenant ID](/img/docs/lifecycle/authenticate-users/identity-providers/enterprise/azure-openid-connect-client-id.png)

Once the application has been created, note the `Application (client) Id` and the `Directory (tenant) Id`.  These will be used respectively as the <InlineField>Client Id</InlineField> value and to construct the <InlineField>Issuer</InlineField> value in your FusionAuth OpenID Connect Identity Provider configuration.

### Configure Permissions

To allow FusionAuth to reconcile user data (such as groups or additional profile information), you may need to add API permissions to your application registration.

Navigate to <Breadcrumb>Identity -> Applications -> App registrations -> Your Application -> API permissions</Breadcrumb>. By default, `User.Read` is usually present. If you plan to use a lambda to map groups, you may need to add `GroupMember.Read.All` or similar permissions, depending on your requirements. Remember to click <InlineUIElement>Grant admin consent for ...</InlineUIElement> after adding any permissions that require it.

### Create A New Microsoft Entra ID Application Secret

Navigate to <Breadcrumb>Identity -> Applications -> App registrations -> Your Application -> Certificates & secrets -> + New client secret</Breadcrumb> to create a new Microsoft Entra ID Application Client Secret.

![Microsoft Entra ID Client Secret](/img/docs/lifecycle/authenticate-users/identity-providers/enterprise/azure-openid-connect-client-secret.png)

Note the `VALUE` of the created client secret.  This will be used as the <InlineField>Client secret</InlineField> value in your FusionAuth OpenID Connect Identity Provider configuration.

### Configure A New FusionAuth OpenID Connect Identity Provider

To create a Microsoft Entra ID Identity Provider return to FusionAuth and navigate to <Breadcrumb>Settings -> Identity Providers</Breadcrumb> and click <InlineUIElement>Add provider</InlineUIElement> and select <InlineUIElement>OpenID Connect</InlineUIElement> from the dialog.

This will take you to the `Add OpenID Connect` panel, and you'll fill out the required fields.

You will need to set the <InlineField>Client authentication method</InlineField> to `HTTP Basic authentication (client_secret_basic)`.

<InlineField>Client Id</InlineField> and <InlineField>Client secret</InlineField> values reference the previously noted Microsoft Entra ID Application's `Application (client) ID`, client secret `VALUE`. The <InlineField>Redirect URL</InlineField> is read only and generated for you based upon the URL of FusionAuth, this value should match the one you configured in your Entra ID application.

Microsoft Entra ID has implemented a well-known configuration endpoint, so FusionAuth will be able to discover the necessary endpoints using a discovery document by entering the Microsoft Authority URL in the <InlineField>Issuer</InlineField> field. To see the <InlineField>Issuer</InlineField> field, you may need to toggle <InlineField>Discover endpoints</InlineField>.

Microsoft Entra ID provides two versions of its OIDC endpoints. **The v2.0 endpoint is recommended** as it supports both work/school accounts and personal Microsoft accounts, and is more compliant with the OpenID Connect specification.

The Microsoft URL may differ across national clouds, so you will need to review the Microsoft documentation to ensure you have the correct URL for your region. For the Microsoft global Microsoft Entra ID service, the URLs are as follows, where `{tenantId}` is the `Directory (tenant) Id` previously noted while creating our Microsoft Entra ID Application.

- Microsoft Entra ID v2 (Recommended): `https://login.microsoftonline.com/{tenantId}/v2.0`
- Microsoft Entra ID v1: `https://login.microsoftonline.com/{tenantId}`

<Aside type="note">
When using the v2.0 endpoint, ensure your <InlineField>Scope</InlineField> field includes `openid`. You may also want to include `profile` and `email` to receive more user information in the initial token.
</Aside>

You may set a <InlineField>Reconcile lambda</InlineField> to map attributes from the JWT provided by Microsoft Entra ID to the FusionAuth user or registration. This is optional. [Learn more about lambdas](/docs/extend/code/lambdas/reconcile/openid-connect-response-reconcile). You may also modify the <InlineField>Button text</InlineField> or <InlineField>Button logo</InlineField> if desired.

![FusionAuth Microsoft Entra ID IdP Configuration](/img/docs/lifecycle/authenticate-users/identity-providers/enterprise/top-of-fusionauth-azure-ad-oidc-config.png)

Scroll down and make sure you enable this Identity Provider for your application. In the following screenshot you will see that we have enabled this login provider for the `Pied Piper` application and enabled `Create registration`.

![FusionAuth Microsoft Entra ID IdP Configuration](/img/docs/lifecycle/authenticate-users/identity-providers/enterprise/bottom-of-fusionauth-azure-ad-oidc-config.png)

That's it, now the <InlineUIElement>Login with Microsoft Entra ID</InlineUIElement> button will show up on the login page of our `Pied Piper` application.

## Combining With SCIM Provisioning

If you are using [SCIM](/docs/lifecycle/migrate-users/scim/) to provision users from Microsoft Entra ID to FusionAuth, those users will already exist in FusionAuth before they attempt their first login.

In this scenario, the following configuration is recommended:

- <InlineField>Linking strategy</InlineField>: Set to `Link on username. Do not create the user if they do not exist.`
- <InlineField>Create registration</InlineField>: Ensure this is enabled for the appropriate application(s) in the Identity Provider configuration.
- <InlineField>User registration</InlineField>: Ensure this is enabled in the <Breadcrumb>Applications -> Your Application -> Registration</Breadcrumb> tab.

Using the `Link on username` strategy ensures that the user's Microsoft username is used to find the existing user provisioned via SCIM. Using a `Do not create` strategy prevents users being created which are intentionally not synced (excluded) through SCIM.
